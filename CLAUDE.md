# Windsor Docs — Claude Code Context

This repo is **vanilla content**, not code. The markdown here is vendored
into [windsorcli.github.io](https://github.com/windsorcli/windsorcli.github.io)
and rendered as the public docs site. Low-level CLI and blueprint
*reference* docs live in the `windsorcli/cli` and `windsorcli/core`
repos under `docs/reference/`. This repo holds the **general** docs:
concepts, how-tos, getting-started, and platform overviews.

Skills live in `.claude/skills/`, which Claude Code discovers
automatically. Other tools (Cursor, agent SDKs) can be pointed at the
same files; the directory is plain markdown with no Claude-specific
metadata.

## What good looks like

1. **Frontmatter is required.** Every page needs `title:` and
   `description:` (≤160 chars). CI rejects pages missing either, or
   with overlong descriptions — both feed into OG tags, search
   snippets, and `llms.txt`.

2. **Link by whether the target file lives in this repo.** If the
   page **is in this repo**, use a **relative `.md` path** —
   `[first project](../getting-started/first-project.md)`. It resolves
   on GitHub and for agents fetching the raw file, and the website
   rewrites it to a clean route (`/getting-started/first-project`) when
   it vendors the docs. If the page **isn't here** — `cli`/`core`
   reference, or anything else off-repo — link the **`windsorcli.dev`
   URL** directly: `[the up command](https://www.windsorcli.dev/reference/cli/commands/up)`.
   It works on GitHub by going to the live site, and the website
   localizes it to a root-relative path at vendor time. Don't use bare
   site paths (`/blueprints/schema`) — they're dead links when the raw
   `.md` is read on its own. Use the host exactly as
   `https://www.windsorcli.dev` — that's the literal the vendor step
   strips, defined once in `windsorcli.github.io/site.config.mjs`; a bare
   `windsorcli.dev` won't localize. (The transform lives in
   `windsorcli.github.io/scripts/vendor-docs.mjs`.)

3. **Pages stand alone.** Agents, search results, and `Open in
   Claude`-style links fetch single pages. No `as we'll see below`
   or `as discussed above` — the section the reader came from may not
   be there. Vale's `Windsor.ForwardReferences` rule flags these.

4. **Plain markdown only.** No MDX components, no JSX, no Astro
   shortcodes. The website's renderer supports them; an agent
   fetching the raw `.md` does not.

5. **Stable headings.** Renaming `## Anatomy` to `## How it works`
   breaks every deep-link to `#anatomy`. Treat heading text as part
   of the page's public API.

6. **No marketing words or future tense.** `Windsor.MarketingWords`
   rejects `seamlessly`, `powerful`, `leverages`, etc.
   `Windsor.FutureTense` flags `will support X` — link an issue
   instead.

7. **Code fences declare a language.** markdownlint MD040 enforces.

Run quality checks before pushing — `markdownlint`, `vale`, `cspell`,
`lychee`. See [CONTRIBUTING.md](CONTRIBUTING.md) for the local
commands.

## Skills

- [`docs-style`](.claude/skills/docs-style/SKILL.md) — voice, structure,
  frontmatter, link conventions, mermaid usage. Read before writing.
- [`docs-review`](.claude/skills/docs-review/SKILL.md) — pre-PR pass
  over a page or diff.
- [`docs-validation`](.claude/skills/docs-validation/SKILL.md) — run
  real `windsor` commands against documented scenarios. Batch
  assertions inside one lifecycle; use before publishing a how-to.
- [`upstream-history`](.claude/skills/upstream-history/SKILL.md) —
  investigate `windsorcli/cli` and `windsorcli/core` history before
  writing or revising claims about behavior.

## Downstream flow

When this repo's `main` is tagged, the website's
[vendor-docs](https://github.com/windsorcli/windsorcli.github.io/blob/main/scripts/vendor-docs.mjs)
script downloads the tarball and mounts these pages into its content
collections. The website owns the final URL scheme — pages from this
repo's `content/` tree land at the site root, alongside reference docs
from the `cli/` and `core/` repos. For local development, the website's
`npm run dev` symlinks straight into this repo so edits show up
immediately.

## What this repo does NOT hold

- CLI reference (commands, flags, config schema) — lives in
  `windsorcli/cli` under `docs/reference/cli/`.
- Blueprint reference (add-ons, modules, schema for `core`) — lives in
  `windsorcli/core` under `docs/reference/core/`.
- Release notes — generated by release-drafter on each repo.
- Marketing / landing pages — those live in the website repo's
  `src/pages/`, not in content collections.
